# apps.mobile — подписанные эндпоинты мобильного приложения

Read-only API для iOS/Android-приложения. Без регистрации пользователей:
приложение аутентифицирует **само себя** общим секретом и HMAC-подписью каждого
запроса. Секрет выбирается по `key_id`: сервер держит JSON-карту key-id → секрет
(`MOBILE_APP_KEYS`), а клиент шлёт заголовок `X-App-Key-Id` (напр. `android-v1`),
по которому сервер находит нужный секрет и проверяет подпись. Смонтировано на
`/app/*`, `label = "mobile"`, самодостаточно — не трогает `/api/`, `donate`,
`website`.

Детали схемы подписи (канонная строка, заголовки `X-App-*` включая
`X-App-Key-Id`, `SignedAppPermission`, replay-окно `±MOBILE_APP_TS_WINDOW`,
`AppInstall`-статистика) — в `signing.py` и `permissions.py`; краткая сводка
инвариантов — в разделе **apps.mobile** корневого `CLAUDE.md`. Этот документ
про **то, как приложение обновляет данные в фоне**.

## Ротация секрета (`MOBILE_APP_KEYS` + `X-App-Key-Id`)

`MOBILE_APP_KEYS` — JSON-карта `{"key-id": "секрет"}`, распарсенная один раз при
старте процесса; проверка делает in-memory `dict.get(key_id)` (без БД/IO на
горячем пути). Несколько ключей активны одновременно → секрет ротируется без
мгновенной поломки старых сборок. Пустая/некорректная карта → fail-closed (каждый
запрос `403`). Канонная строка **не меняется**: подделанный `key_id` выберет
другой секрет, подпись не сойдётся — key-id неявно связан подписью.

`AppInstall.key_id` (миграция `0003`) хранит последний key-id каждого инстала —
перед снятием старого ключа админ через фильтр в админке убеждается, что ни один
инстал его больше не репортит. Рунбук ротации: добавить новый ключ в
`MOBILE_APP_KEYS` → выпустить сборку приложения с новым секретом и key-id →
дождаться, когда ни один `AppInstall` не репортит старый key-id → убрать старый
ключ из карты.

## 403-трекинг (`AppAuthFailure`)

Каждый провалившийся запрос к `/app/*` логируется (`WARNING`) и записывается
**агрегированной** строкой `AppAuthFailure` с ключом `(ip, key_id, reason)` —
не по одной строке на попытку, чтобы брутфорс не переполнял таблицу. Коды
причины: `no_keys`, `missing_headers`, `unknown_key`, `bad_ts`, `expired_ts`,
`bad_sig`.

`SignedAppPermission` делает только crypto-проверку и заполняет
`request.app_denial` — **без DB-записей**. Лог и запись в БД живут в
`AppAPIView.permission_denied()` (DRF-хук, вызывается до `_record_install`).
Отклонённый запрос **никогда** не создаёт строку `AppInstall`. Клиенту
возвращается нейтральный `403 {"detail": "Forbidden"}` без подсказки о
причине. Таблица читается через read-only `AppAuthFailureAdmin` в Django admin
(фильтры по `reason`, `key_id`; сортировка по `count` убыванием).

## Что обновляется

Приложение держит локальную копию справочников и в фоне поддерживает её
актуальной:

- **список гонок** — какие гонки есть, их статус/флаги;
- **команды гонки** — состав участников;
- **легенда гонки** — контрольные точки; легенда **запертых** КП зашифрована и
  расшифровывается приложением **офлайн только после физического сканирования**
  NFC-кода КП (см. раздел **Шифрование легенды**). Сырой `nfc_uid` на провод не
  уходит; вместо него — единый идентификатор `code`. На каждый физический тег
  отдаётся запись в `tags`: **идентичность** (`bid → point`, всегда, и для
  открытых КП тоже) и — только у запертых КП — **конверт-расшифровка** (`iv`/`ct`).

## Шифрование легенды (запертые КП)

Легенда **запертой** КП (`Checkpoint.is_legend_locked=True`) скрывает
`cost`/`description` за конвертным шифрованием, чтобы приложение могло
расшифровать их **только офлайн и только после физического сканирования NFC-кода
этой КП**. Шифрование **выборочное** (флаг на каждую КП), и один NFC-код может
открывать **настраиваемое подмножество** КП (пересечения разрешены).

**Модель угроз:** противник — само приложение в руках участника. Утечка БД —
**вне области**: `cost`/`description` могут лежать в БД в открытом виде. Поэтому
**at-rest-шифрования и мастер-ключа нет** — шифруется только то, что уходит с
сервера.

### Схема (конвертное шифрование, AES-256-GCM + HKDF)

Примитивы — `crypto.py` (`seal`/`unseal`/`derive_wrap_key`; `unseal`, а не
`open`, чтобы не затенять builtin и не ронять flake8).

1. Каждая запертая КП запечатывается собственным случайным `content_key`
   (32 B) в модель `CheckpointSecret` (`O2O → Checkpoint`,
   `related_name="secret"`): `enc_blob = {"iv","ct"}` — AES-GCM от
   `{cost, description}`, `aad = str(cp.id)`. `id`/`number`/`type` остаются
   публичными. Секрет существует **только** у запертых КП.
2. Каждый `CheckpointTag` несёт случайный `code` (16 B), записываемый в
   user-память NFC-метки **отдельным тулингом** (вне этого кода) для **всех**
   тегов — и запертых, и открытых, и `bid = sha256(code).hexdigest()[:16]`
   (16 hex-символов; мобильный клиент обязан считать его идентично).
   `bid → point` (`point_id` тега) — это **идентичность**: офлайн-распознавание
   `cp_id` для любой КП. Дополнительно у тега запертой КП есть пер-тег
   `bundle_blob = {"iv","ct"}` — AES-GCM от `{cp_id: content_key}` по
   набору `unlocks` тега, ключ обёртки `= HKDF(code)`, `aad = bid`. Пустой
   `unlocks` ⇒ тег открывает свою же КП (`[point]` по умолчанию в
   `build_bundle`). У открытого тега `bundle_blob` нет (`iv`/`ct` → `None`).
3. Офлайн-поток приложения: прочитать `code` → `bid = sha256(code).hexdigest()[:16]` →
   найти запись по `bid` в `tags` → `point` даёт `cp_id` (идентичность). Если у
   записи есть `iv`/`ct` (запертая КП), `HKDF(code)` расшифровывает бандл →
   `{cp_id: content_key}` → расшифровать `enc` каждой запертой КП.

Конверт (косвенность через `content_key`) и даёт «код → подмножество КП» с
пересечениями (одна КП достижима из нескольких кодов, один код открывает
несколько КП) без пере-шифровки КП под каждый код.

### Хранение и консистентность

Шифртекст и бандлы **предвычислены и хранятся в БД** (так ETag стабилен).
`content_key`/`code` хранятся сырыми (БД доверенная). Согласованность блобов
держат **сигналы** (`signals.py`): `post_save(Checkpoint)` пере-печатывает
секрет, а при переключении флага запертости пересобирает бандлы
`cp.tags.all() ∪ cp.unlocked_by.all()`; `post_save(CheckpointTag)` и
`m2m_changed(unlocks)` пересобирают бандл тега. Защита от рекурсии:
`post_save(CheckpointTag)` рано выходит, когда `update_fields` ровно
`{"code","bid","bundle_blob","updated_at"}` (sentinel сервисного слоя), а
`m2m_changed`-путь огорожен thread-local-флагом.

Массовые операции **обходят сигналы**, поэтому есть команды:

- `manage.py rebuild_legend_crypto [--race <id>] [--regenerate-codes]` —
  бэкфилл после миграции + ремонт (пере-печатать все запертые КП, пересобрать
  все бандлы); `--regenerate-codes` форсит новые `code` (пере-провижининг).
- `manage.py export_legend_codes --race <id>` — печатает `nfc_uid / номер КП /
  code(hex)` для записи кодов в метки.

Bulk-экшены админки «Запереть/Открыть легенду» и «Пересобрать бандл» **обязаны
итерировать и `save()`/звать сервис**, а не `queryset.update()` — иначе сигналы
пропускаются, `CheckpointSecret` не создаётся, и сериализатор отдаёт `cost`/
`description` в открытом виде.

### Что отдаётся

`LegendCheckpointSerializer` ветвится: запертая КП → `{id, number, type, enc}`;
открытая → `{id, number, type, cost, description}`. `TagSerializer` отдаёт на
**каждый** тег `{bid, point, check_method}` (идентичность — всегда, и для
открытых КП) и `iv`/`ct` из `bundle_blob` (только у запертых КП; у открытых →
`None`). **`tag_hash` убран** — офлайн-идентичность теперь `bid → point` для
любого тега; путь скоринга при онлайн-сканировании живёт в `api` и матчит по
`nfc_uid` (без изменений).

```json
{
  "race": 42,
  "checkpoints": [
    {"id": 153, "number": 1, "type": "kp", "enc": {"iv": "...", "ct": "..."}},
    {"id": 154, "number": 2, "type": "kp", "cost": 4, "description": "Дерево у дороги"}
  ],
  "tags": [
    {"bid": "a1b2...", "point": 153, "check_method": "offline", "iv": "...", "ct": "..."},
    {"bid": "c3d4...", "point": 154, "check_method": "offline"}
  ]
}
```

## Принцип синхронизации: полный payload + условный GET (ETag / If-None-Match)

Мы **сознательно не делаем дельта-синк** (`?since=<ts>`, присылать только
изменившиеся строки). На наших объёмах (десятки КП, сотни команд на гонку) он не
окупается: требует `updated_at` на всех моделях **и** тумбстоунов удалений
(удалённая строка не присылает изменений — без отдельной пометки клиент о ней не
узнает). Это лишняя схема синхронизации.

Вместо этого каждый ресурс отдаётся **целиком**, а дешевизну фонового опроса даёт
встроенный в HTTP механизм условных запросов — `ETag` + `If-None-Match`.

### ETag — отпечаток версии ресурса

На каждый ответ сервер вешает заголовок `ETag` — строку-«версию» текущего
содержимого. Правило одно: **тело изменилось → ETag другой; тело то же → ETag
тот же.**

Считается дёшево, без сериализации данных:

- **teams**: `hash(MAX(Team.updated_at), COUNT(*))` по командам гонки **плюс**
  агрегаты по участникам (`Athlet`) и категориям (`Category`) — в ответ teams
  встроен и список категорий гонки, поэтому переименование/реордер/добавление/
  удаление категории тоже двигает версию (`MAX(Category.updated_at)` +
  `COUNT`). Добавили команду → вырос `COUNT`; отредактировали → сдвинулся
  `MAX(updated_at)`.
- **legend**: `MAX(updated_at), COUNT` по `Checkpoint` **плюс** по
  `CheckpointSecret` и `CheckpointTag` того же hidden-исключённого набора,
  поэтому запирание/отпирание КП, пере-шифровка и правка
  тега двигают версию. **Build-независим** — без `key_id` (см. **Статус** ниже).
- **races**: аналогично по `MAX(updated_at), COUNT` соответствующих строк.

ETag оборачивается в кавычки (часть синтаксиса по RFC) и сравнивается
байт-в-байт: `ETag: "a1b2c3"`.

### Поток запросов

**1. Первый запрос — у клиента нет кеша:**

```
→ GET /app/race/8/teams/
   X-App-Sig: …

← 200 OK
   ETag: "a1b2c3"
   {"race": 8, "categories": [ …категории гонки… ], "teams": [ …весь список… ]}
```

Клиент сохраняет локально тело **и** строку `"a1b2c3"`.

Блок `categories` едет **внутри** ответа teams (отдельного эндпоинта нет):
каждая категория — `id`/`code`/`short_name`/`name`/`order`, отсортированы по
`order, id`. Включаются **все** категории гонки, в том числе `is_active=False`
(команда может ссылаться на снятую категорию — её `category2`-id должен
резолвиться); само поле `is_active` не отдаётся. Изменение списка категорий
двигает тот же `versions.teams` / ETag teams — отдельного `versions.categories`
нет.

**2. Фоновый опрос — ничего не поменялось:**

```
→ GET /app/race/8/teams/
   If-None-Match: "a1b2c3"      ← эхом возвращаем запомненный ETag
   X-App-Sig: …

← 304 Not Modified
   (тело пустое)
```

Сервер посчитал текущий ETag, он совпал → `304` **без тела**. Копеечный опрос:
сотни байт заголовков, ноль данных, ноль парсинга JSON на телефоне. Главное —
при 304 сервер **даже не сериализует** список: посчитал ETag (один SQL
`aggregate`) и вышел.

**3. Кто-то добавил/изменил команду:**

```
→ GET /app/race/8/teams/
   If-None-Match: "a1b2c3"
   X-App-Sig: …

← 200 OK
   ETag: "d4e5f6"               ← новая версия
   {"race": 8, "categories": [ … ], "teams": [ …обновлённый список… ]}
```

ETag не совпал → полный свежий payload + новый ETag. Клиент перезаписывает копию
и запоминает `"d4e5f6"`.

### Почему удаления «просто пропадают»

При смене ETag отдаётся **весь актуальный список**. Удалённой команды в новом
списке нет → клиент затирает старую копию новой, строка исчезает сама. Никаких
тумбстоунов — отсутствие дельта-синка и есть отсутствие схемы синхронизации.

### Серверная сторона (псевдокод)

```python
def get(self, request, race_id):
    current_etag = self.compute_etag(race_id)        # hash(MAX(updated_at), COUNT)
    if request.headers.get("If-None-Match") == current_etag:
        return Response(status=304)                  # данные НЕ сериализуем
    data = self.serialize(race_id)
    resp = Response(data)
    resp["ETag"] = current_etag
    return resp
```

## Взаимодействие с подписью

- `If-None-Match` **не входит** в канонную строку
  (`method\nfull_path\nts\nsha256(body)`) и не должен: он не меняет смысл запроса.
  В худшем случае при рассинхроне клиент получит `200` вместо `304` и просто
  перекачает данные — безопасность не страдает, `signing.py` трогать не нужно.
- Порядок проверок: **подпись → ETag**. `SignedAppPermission` отрабатывает как
  обычно (невалидная подпись → нейтральный `403`); `304` возвращается **только
  после** успешной проверки подписи, на валидном запросе. 304 — это не ошибка и
  не разновидность 403.
- Используем **сильные** ETag (без префикса `W/`): отпечаток привязан к данным.

## Два сервера и хэндофф

В бою два сервера: **облачный** (в интернете) и **локальный** (в LAN на гонке).
Данные правят на **обоих** — например, во время гонки админ на локальном
уменьшает состав команды (участник не приехал). Опасность: телефон ловит
интернет, синкается с облака и **возвращает** прежний состав → команде не хватает
людей для отметки на КП.

Это **не** задача свежести кеша — это **источник правды и разрешение
конфликтов**. ETag/`data_version` отвечают на «актуальна ли моя копия *этого
сервера*», но **не** на «чьи данные главнее, когда серверы разошлись».

### Версия конфликт не решает (и может усугубить)

Версия арбитрит конфликт, только если её выдаёт **один** авторитет. У нас два
независимых писателя — счётчики версий у серверов свои, **сравнивать их между
origin нельзя**. Правило «побеждает бóльшая версия» → телефон увидит у облака
версию повыше (там состав не трогали) и вернёт старое число. Конфликт двух
одновременных писателей надёжно решается не версией, а тем, чтобы **их не было
одновременно**.

### Правило: один авторитет на гонку в каждой фазе

```
До гонки:    источник правды — облако.  Локальный — зеркало (тянет с облака).
Во время:    источник правды — ЛОКАЛЬНЫЙ. Телефон читает операционные данные
             этой гонки ТОЛЬКО с локального. Облако к ней не допускается.
После:       сверка локальный → облако (backend-задача, телефона не касается).
```

```
   облако ──(сид перед гонкой)──▶ локальный
                                     │  во время: правки админа + отметки КП
                                     ▼
   облако ◀──(сверка после гонки)── локальный
```

Во время гонки телефон **вообще не тянет состав этой гонки из облака**, поэтому
перезаписать локальную правку нечем — писатель один, мержить нечего.

### Как это выражено в данных

1. **Origin-тег на каждой кешированной записи** — телефон помнит, откуда взял
   данные: `source: "local"` / `source: "cloud"`.
2. **Лиз авторитета в payload.** Локальный сервер помечает гонку как оперируемую
   им — **со сроком**, а не просто флагом:

   ```json
   { "data_source": "local", "lease_expires_at": 1718200000 }
   ```

   где `lease_expires_at` ≈ конец гонки + запас. Телефон, увидев в LAN такой
   сервер с этой гонкой, **пинит** её на локальный: состав/легенда и ETag —
   только локального.
3. **Правило «не понижать через границу origin»:** запись с `source:"local"` для
   активного лиза **никогда** не перезаписывается данными `source:"cloud"`, даже
   если телефон случайно дёрнул облако.

### Возврат на cloud: по истечению лиза, а не по потере связи

Пин — это **лиз, а не залипающая защёлка**. Освобождает его **время/фаза, а не
связь**. Триггер «вернуться на cloud, когда локальный недоступен» — **ошибка**:
телефон вышел из Wi-Fi на 5 минут → откат на cloud → cloud вернул прежний состав.

- **Транзиентный обрыв во время гонки** → лиз ещё жив → телефон работает с
  последней локальной копией из кеша, в облако за этой гонкой не ходит.
- **Гонка кончилась, локальный выключили** → лиз сам истёк → телефон
  автоматически возвращается на cloud, без «чистого» сигнала от мёртвого сервера.

**Heartbeat:** каждый успешный синк с локальным **продлевает**
`lease_expires_at`. Телефон уходит на cloud, только когда **и** не может продлить
(локальный недоступен), **и** последний известный срок истёк.

**Быстрый путь штатного завершения:** если телефон дотянулся до локального, а тот
говорит `data_source:"cloud"` / `live:false` (или убрал гонку) — пин снимается
**сразу**.

Триггеры возврата на cloud:

| Событие | Возврат на cloud? |
|---|---|
| Локальный недоступен, лиз ещё жив | **Нет** (используем кеш локального) |
| Локальный недоступен, лиз истёк | Да (авто) |
| Локальный отвечает `live:false` / `data_source:"cloud"` | Да (сразу) |
| Лиз истёк, но локальный ещё отдаёт live и продлевает | Нет (heartbeat продлил) |

Лиз — **per-race**: одна гонка может быть live на локальном, другая — обычная
облачная.

**Caveat:** между истечением лиза и сверкой `локальный → облако` есть окно, где
телефон может показать ещё несведённые облачные данные. Ставь `lease_expires_at`
**с запасом на сверку** — лиз должен пережить момент, когда облако догонит
локальные правки (либо постгоночная сверка пушит в облако до истечения лиза).

### Где остаётся ETag / `data_version`

Там же — **внутри одного origin**, для свежести: ETag локального проверяет
актуальность локальной копии, ETag облака — облачной. Это по-прежнему **два ETag,
по origin** (см. выше). Межсерверный арбитр — флаг авторитета
(`data_source`/`lease_expires_at`), **а не** число версии.

## Манифест: эндпоинт `sync`

Фоновый опрос трёх ресурсов (`races`/`teams`/`legend`) по отдельности — это три
подписанных запроса на каждый тик. Чтобы схлопнуть их в **один**, есть лёгкий
эндпоинт-манифест:

```
GET /app/race/<id>/sync/        → mobile:sync
```

Отдаёт по строке-версии на каждый ресурс гонки **плюс** флаг авторитета — то есть
объединяет две вещи, о которых шла речь выше: указатели свежести (для ETag-логики)
и лиз хэндоффа (для выбора сервера):

```json
{
  "race": 8,
  "data_source": "local",
  "lease_expires_at": 1718200000,
  "versions": {
    "teams":  "a1b2c3",
    "legend": "d4e5f6"
  }
}
```

Поток в фоне:

1. Телефон дёргает **один** `sync` (дёшево: пара SQL-`aggregate`, без сериализации
   данных).
2. Сверяет `versions[*]` с тем, что закешировал; тянет полным запросом **только**
   изменившиеся ресурсы.
3. Из того же ответа берёт `data_source`/`lease_expires_at` — продлевает лиз и
   решает, с каким сервером вообще работать (см. **Два сервера и хэндофф**).

Так фоновый тик — это `1` запрос вместо `3`, а полные `GET teams/legend` уходят
только по факту изменения. `data_source`/`lease_expires_at` приходят из манифеста
**того сервера, к которому обратились**: локальный объявляет свой лиз, облако — что
гонка облачная.

Почему манифест, а не просто ETag/304 на каждом ресурсе: на трёх ресурсах выигрыш
по числу запросов скромный, но манифест даёт **консистентный снимок** версий +
авторитета за один round-trip и одно место для `lease_expires_at`. Базовый
ETag/`If-None-Match` на ресурсах остаётся (см. выше) — `sync` его не заменяет, а
надстраивается: манифест говорит «что качать», ETag на ресурсе остаётся защитой от
гонки между манифестом и полным GET.

## Эндпоинты

| Метод | Путь | Имя | Отдаёт |
|-------|------|-----|--------|
| GET | `/app/races/` | `mobile:races` | список гонок (лёгкий: id, slug, name, date, reg_status, флаги `is_*`) |
| GET | `/app/race/<id>/teams/` | `mobile:teams` | команды гонки + категории (`categories`) |
| GET | `/app/race/<id>/legend/` | `mobile:legend` | КП (запертые — только `enc`) + пер-тег `tags` (`bid → point` идентичность всегда; `iv`/`ct` только у запертых); легенда published-гонки отдаётся всегда, КП типа `hidden` («Скрытый») исключены |
| GET | `/app/race/<id>/sync/` | `mobile:sync` | манифест: версии ресурсов + `data_source`/`lease_expires_at` |

Все — наследники `AppAPIView` (подпись + `AppInstall`-статистика); у `races`,
`teams` и `legend` — поддержка `ETag`/`If-None-Match` (304), у `sync` — лёгкий
манифест версий для фонового обновления (без 304). Версия `races` глобальная
(`races_version()`, без `race_id`) и сознательно **не входит** в пер-гоночный
манифест `sync` — список гонок клиент пробует напрямую своим conditional GET
(это входная точка приложения, до выбора `race_id`).

> **Статус:** реализованы `mobile:races` (с `ETag`/`If-None-Match` → 304;
> отпечаток — `MAX(Race.updated_at)|COUNT` по published-гонкам, поле
> `Race.updated_at` — миграция `0079`), `mobile:legend` (с `ETag`/`If-None-Match`
> → 304; запертые КП отдают только `enc`, идентичность тегов — в `tags`),
> `mobile:teams` (с `ETag`/`If-None-Match` → 304) и
> `mobile:sync`. Отпечаток `teams` считается по
> `MAX(Team.updated_at)`/`MAX(Athlet.updated_at)`/`COUNT` команд и участников
> **плюс** `MAX(Category.updated_at)`/`COUNT(Category)` по категориям гонки
> (поле `Athlet.updated_at` добавлено, чтобы переименование участника сдвигало
> версию; `Category.updated_at` — миграция `0078` — чтобы правка встроенного в
> ответ списка категорий тоже двигала версию). Отпечаток `legend` —
> `MAX(Checkpoint.updated_at)|COUNT` по **hidden-исключённому**
> набору КП (поле `Checkpoint.updated_at` добавлено миграцией `0077` по той же
> причине, что и `Athlet.updated_at` — ловить правки на месте) **плюс** ещё два
> агрегата `MAX(updated_at)|COUNT` по тому же hidden-исключённому набору:
> `CheckpointSecret` (пере-шифровка / появление-исчезновение `enc` при
> запирании-отпирании) и `CheckpointTag` (`code`/`unlocks`/`bundle`/`check_method`).
> Легенда теперь **build-независимая**: шифртекст и конверты **предвычислены и
> хранятся в БД** (не зависят от секрета сборки), поэтому отпечаток **не** содержит
> `key_id` и две разные сборки получают **один** ETag/`versions.legend` — ротация
> `X-App-Key-Id` больше не пере-качивает легенду. Манифест `sync` несёт **оба**
> `versions.teams` и `versions.legend`. Лиз-хэндофф **застаблен**: `data_source`
> берётся из настройки `MOBILE_DATA_SOURCE` (по умолчанию `"cloud"`),
> `lease_expires_at` всегда `null` — реальный per-race лиз/хэндофф остаётся целевой
> схемой, описанной выше.
