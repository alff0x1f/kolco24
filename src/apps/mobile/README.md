# apps.mobile — подписанные эндпоинты мобильного приложения

Преимущественно read-only API для iOS/Android-приложения. Без регистрации
пользователей **на уровне сборки**: приложение аутентифицирует **само себя**
общим секретом и HMAC-подписью каждого запроса. Секрет выбирается по `key_id`:
сервер держит JSON-карту key-id → секрет (`MOBILE_APP_KEYS`), а клиент шлёт
заголовок `X-App-Key-Id` (напр. `android-v1`), по которому сервер находит нужный
секрет и проверяет подпись. Смонтировано на `/app/*`, `label = "mobile"`.
Read-путь самодостаточен — не трогает `/api/`, `donate`, `website`; **исключения
— загрузка GPS-треков и взятий КП** (`POST /app/race/<id>/track/` и
`POST /app/race/<id>/marks/`): их модели `TrackPoint`/`Mark`+`MarkPresent`
держат FK в `website.Team`/`website.Race` (cross-app FK, как у `apps.race`).

Поверх подписи сборки есть **тонкий слой записи** — вход по человеку
(`POST /app/login/` → отзываемый bearer-токен) и добавление тегов легенды
(`POST /app/race/<id>/tags/`). Запись надстраивается, а не заменяет read-only
контракт: см. раздел **Запись: вход по человеку и добавление тегов легенды**.
Загрузка GPS-треков (`POST /app/race/<id>/track/`) и взятий КП
(`POST /app/race/<id>/marks/`) — **третий и четвёртый POST**, но гейтятся
**только подписью сборки** (без bearer-токена); см. разделы
**Загрузка GPS-треков** и **Загрузка взятий КП**.

Детали схемы подписи (канонная строка, заголовки `X-App-*` включая
`X-App-Key-Id`, `SignedAppPermission`, replay-окно `±MOBILE_APP_TS_WINDOW`,
`AppInstall`-статистика) — в `signing.py` и `permissions.py`; краткая сводка
инвариантов — в разделе **apps.mobile** корневого `CLAUDE.md`. Этот документ
про **то, как приложение обновляет данные в фоне**.

## Ротация секрета (`MOBILE_APP_KEYS` + `X-App-Key-Id`)

`MOBILE_APP_KEYS` — JSON-карта `{"key-id": "секрет"}`, распарсенная один раз при
старте процесса; проверка делает in-memory `dict.get(key_id)` (без БД/IO на
горячем пути). Несколько ключей активны одновременно → секрет ротируется без
мгновенной поломки старых сборок. Пустая/некорректная карта → fail-closed (каждый
запрос `403`). Канонная строка **не меняется**: подделанный `key_id` выберет
другой секрет, подпись не сойдётся — key-id неявно связан подписью.

`AppInstall.key_id` (миграция `0003`) хранит последний key-id каждого инстала —
перед снятием старого ключа админ через фильтр в админке убеждается, что ни один
инстал его больше не репортит. Рунбук ротации: добавить новый ключ в
`MOBILE_APP_KEYS` → выпустить сборку приложения с новым секретом и key-id →
дождаться, когда ни один `AppInstall` не репортит старый key-id → убрать старый
ключ из карты.

## 403-трекинг (`AppAuthFailure`)

Каждый провалившийся запрос к `/app/*` логируется (`WARNING`) и записывается
**агрегированной** строкой `AppAuthFailure` с ключом `(ip, key_id, reason)` —
не по одной строке на попытку, чтобы брутфорс не переполнял таблицу. Коды
причины: `no_keys`, `missing_headers`, `unknown_key`, `bad_ts`, `expired_ts`,
`bad_sig`.

`SignedAppPermission` делает только crypto-проверку и заполняет
`request.app_denial` — **без DB-записей**. Лог и запись в БД живут в
`AppAPIView.permission_denied()` (DRF-хук, вызывается до `_record_install`).
Отклонённый запрос **никогда** не создаёт строку `AppInstall`. Клиенту
возвращается нейтральный `403 {"detail": "Forbidden"}` без подсказки о
причине. Таблица читается через read-only `AppAuthFailureAdmin` в Django admin
(фильтры по `reason`, `key_id`; сортировка по `count` убыванием).

## Запись: вход по человеку и добавление тегов легенды

Read-эндпоинты аутентифицируют **сборку**. Запись требует знать **кто** именно
действует (полевой админ привязывает NFC-чип к КП), поэтому поверх подписи сборки
лежит **слоистый стек прав**: подпись сборки → личность → авторизация на гонку.

```
POST /app/login/                 SignedAppPermission                      → bearer-токен
POST /app/logout/                SignedAppPermission + IsMobileUser       → revoke
POST /app/race/<id>/tags/        SignedAppPermission + IsMobileUser
                                   + CanEditRaceLegend                    → CheckpointTag
```

**Scope — только ADD:** добавление тега. Нет редактирования/удаления тегов,
правки КП, офлайн-очереди и клиентской криптографии — сервер остаётся
единственным источником крипты (теги создаются через `instance.save()`, чтобы
отработали те же сигналы `ensure_code`/`build_bundle`, что и в вебе/`manage.py`).

### Токен (`MobileToken`)

Личная учётка кладётся **поверх** HMAC сборки, а не вместо него: сборка
доказывает «я — наш билд», токен — «я — такой-то человек». Токен —
**непрозрачный, отзываемый, в БД** (не JWT): для админ-доступа приоритет —
мгновенный отзыв (потерянный телефон / снятый из крю), а blocklist у JWT убивает
его же statelessness. Модель `MobileToken` (`src/apps/mobile/models.py`):

- `user` FK, `token_hash` (`unique`), `created_at`, `last_used_at`, `expires_at`,
  `revoked_at`; `@property is_active = revoked_at is None and expires_at > now`.
- Хранится **только** `sha256`-хекс высокоэнтропийного `secrets.token_urlsafe(32)`
  — сырой токен отдаётся клиенту **ровно один раз** при входе. Хеш токена ≠ хеш
  пароля: 256-битный случайный токен не брутфорсится, поэтому быстрый
  несолёный `sha256` корректен (и сильнее DRF authtoken, хранящего плейнтекст);
  поиск идёт по индексированной колонке `token_hash`.
- **Не** ETag-фингерпринтится → нет дисциплины `updated_at`.
- TTL — `MOBILE_TOKEN_TTL = timedelta(days=30)` в `settings.py` (сосед
  `MOBILE_APP_TS_WINDOW`). Хелперы — `tokens.py`: `generate_token() -> (raw,
  token_hash)`, `hash_token(raw)`, `resolve_token(raw)` (проверяет `is_active`,
  best-effort штампует `last_used_at`).

Несколько одновременных токенов на пользователя (строка-на-вход), каждый
отзывается индивидуально.

### `POST /app/login/`

Гейтится **только** `SignedAppPermission` (HMAC сборки), своего токена не
требует — так даже парольный вход достижим только из наших сборок. Логин
**общий**, не админ-специфичный: любой валидный пользователь получает токен,
а админ-способность решается **по-действию** через `can_edit_race` — дверь
открыта под будущие фичи обычных пользователей без переделки. Тело — `{email,
password}` (`LoginSerializer`); аутентификация через
`apps.accounts.backends.EmailBackend` (`email__iexact`), как в вебе.

Ответ: `{token, expires_at}` (200). Ошибки **enumeration-safe**: неверный пароль
и неизвестный email оба → `401` с **одинаковым** сообщением «Неверный email или
пароль», без подсказки. Кривое тело (нет поля / не-JSON) → `400` от сериализатора
(отличается от 401, но тоже ничего не выдаёт о существовании учётки).

### `POST /app/logout/`

Стек `SignedAppPermission + IsMobileUser`. Флипает `revoked_at = now()` на
**предъявленном** токене — другие токены того же пользователя остаются валидны
(отзыв пер-устройство, не пер-аккаунт).

### `POST /app/race/<id>/tags/`

Полный стек `SignedAppPermission + IsMobileUser + CanEditRaceLegend`,
троттлинг `mobile-write`. Тело — `{checkpoint_id, nfc_uid}` (`TagCreateSerializer`), где
**`checkpoint_id` = id КП, а не `number`** (`Checkpoint.number` не уникален в гонке →
риск `MultipleObjectsReturned`; приложение уже знает `id` из синканной легенды).
Пустой `nfc_uid` отбивается сериализатором (400) до модельного `save()`, который
на пустом UID кидает `ValueError` (иначе 500).

Поток: резолв published-гонки (404) → резолв **не-`hidden`** КП по `id` внутри
гонки (404) → нормализация `nfc_uid` (`.strip().upper()`) → идемпотентность/
конфликт → создание `CheckpointTag(checkpoint=cp, nfc_uid=nfc_uid, created_by=request.mobile_user)`
в `transaction.atomic()` с `instance.save()` (ловим `IntegrityError` от
глобального `unique(nfc_uid)` при гонке двойного тапа → перезапрос строки,
идемпотентный 200). Ответ `{bid, checkpoint_id: cp.id, number: cp.number, nfc_uid, code: code.hex()}`
(201, либо 200 на идемпотентном попадании) — `code` приложение пишет в
NFC-user-память чипа.

- тот же `nfc_uid` на **той же** КП → идемпотентный 200 (без дубля строки);
- тот же `nfc_uid` на **другой** КП → `409` (не авто-перепривязываем);
- если у идемпотентной строки `bid == ""`/`code is None` (создана в обход
  сигналов) — она чинится через `build_bundle` до ответа, без `None.hex()` (500).

`CheckpointTag.created_by` — `FK(AUTH_USER_MODEL, null, on_delete=SET_NULL,
related_name="provisioned_tags")` (миграция `website/0088`), проставляется из
`request.mobile_user`. В фингерпринт легенды **не** входит (`legend_version` —
`MAX(updated_at)|COUNT`), но создание тега двигает `versions.legend`/ETag через
`CheckpointTag.updated_at` — без правок `versioning.py`.

### Слой прав (`permissions.py`)

- **`IsMobileUser`** — личность: читает `Authorization: Bearer <token>`, через
  `resolve_token` ставит `request.mobile_user` + `request.mobile_token`. Только
  идентичность, ничего не авторизует. Стек — **после** `SignedAppPermission`.
- **`CanEditRaceLegend`** — авторизация на гонку: читает `view.kwargs["race_id"]`,
  грузит `Race` (нет → 404), делегирует в `apps.race.permissions.can_edit_race`
  (тот же superuser-или-`RaceAdmin(role=ADMIN)`, что у вебового
  `RaceLegendEditView`). `request.mobile_user` читается защитно
  (`getattr(..., None)` → `False`/403, не `AttributeError`/500 при
  перестановке стека). Стек — **после** `IsMobileUser`.

**Нейтральная vs действенная ошибка — намеренный раскол.** Анонимный слой сборки
(`SignedAppPermission`) отдаёт нейтральный `403 {"detail":"Forbidden"}` без
подсказки (чтобы не помогать брутфорсу). Аутентифицированные слои — действенные:
- невалидный/просроченный/отозванный токен → **401**. Тонкость: при
  `authentication_classes = []` поднятые DRF-ные `NotAuthenticated`/
  `AuthenticationFailed` рендерятся как **403** (нет аутентикатора для
  `WWW-Authenticate`), поэтому `IsMobileUser` кидает свой `MobileTokenInvalid`
  (`APIException` с `status_code = 401`);
- юзер без прав на гонку → действенный **403** (дефолтное сообщение DRF).

### Троттлинг

Первое применение в этом приложении — намеренно минимальное. Логин уже за
HMAC-сборки, так что брутфорс «в масштабе» требует валидного секрета сборки;
троттл — второй потолок. Плоский IP-keyed `ScopedRateThrottle` (без сабкласса,
без чтения `request.data`/email внутри троттла — чтобы не зацепить порядок чтения
тела). `DEFAULT_THROTTLE_RATES = {"mobile-login": "5/min", "mobile-write":
"60/min"}` в `REST_FRAMEWORK`; `throttle_scope` ставится **по-вью** (глобального
`DEFAULT_THROTTLE_CLASSES` нет). Превышение → `429`. Троттлингу нужен кеш-бэкенд:
добавлен явный `CACHES` (Django `LocMemCache` — пер-процессный, счётчики
приблизительны под несколькими воркерами, что приемлемо при гейте HMAC-сборки).
Email-keyed троттл сознательно **отложен** (YAGNI при гейте сборки).

### Подпись тела на POST

`build_canonical` подписывает `method\nfull_path\nts\nsha256_hex(body)` — тело
**уже** входит в канонную строку, так что подпись POST структурно работает.
Django буферизует `request.body`, поэтому чтение тела в `SignedAppPermission` до
DRF-парсинга `request.data` **не** роняет `RawPostDataException` (раньше в
`CLAUDE.md` была пометка «подпись тела только для GET» — снята). Подделанное тело
(подпись над устаревшим телом) → нейтральный `403`.

## Загрузка GPS-треков (`POST /app/race/<id>/track/`)

Серверный приёмник GPS-треков, которые Android-приложение шлёт во время гонки
(батчами в фоне, с подписью и ретраями — клиентская часть уже зашита в сборке).
Эндпоинт **только принимает, хранит и подтверждает** точки — организаторская
карта/дашборд, читающий треки обратно, это **отдельная будущая задача, здесь вне
области**.

**Auth — только подпись сборки.** Гейт — дефолтный `AppAPIView`
(`[SignedAppPermission]`), как у read-эндпоинтов. **Не** часть пер-человекового
слоя записи: запись трека идёт с телефонов **участников** (не админов), bearer-
токен не нужен. Граница доверия та же, что у read: валидная сборка может
запостить трек на **любой** `team_id` гонки; `team_id`/`install_id` спуфятся,
принимаются как есть. Троттлинг — `mobile-write` (60/min), как у tag-create.

**Модель `TrackPoint`** (`models.py`, FK cross-app в `website`, как у `apps.race`):
одна строка на GPS-фикс, **PK = сгенерированный клиентом UUID** (`id`). Поскольку
ключ идемпотентности *и есть* первичный ключ, upsert схлопывается в
`bulk_create(objs, ignore_conflicts=True)` — повторно присланная точка ловит
PK-конфликт и молча пропускается (идемпотентный успех), без `update_or_create` и
без вторичного индекса. Модель **write-only / immutable**: нет `updated_at`, она
сознательно **не** в `versioning.py` и не трогает ETag/`sync`-машинерию.

- **Атрибуция устройства:** `install_id` (из `request.app_meta["install_id"]`,
  заголовок `X-Install-Id`) хранится, чтобы команда, пишущая с двух телефонов,
  разделялась: `segment_id` (случайный UUID в проде) различает **сессии записи**,
  `install_id` группирует много сессий одного телефона в «телефон A» vs «B». Без
  изменений в приложении.
- **Вторичного индекса нет.** Приём не делает чтений по `TrackPoint` —
  идемпотентный upsert опирается только на автоматический PK-индекс. Композитный
  индекс под будущую организаторскую карту (вероятно с хвостовой временной
  колонкой для хронологии сегмента) **отложен до задачи read-эндпоинта** — его
  правильная форма не известна, пока нет самого запроса.

**Контракт провода** (точно совпадает с приложением — `TrackDtos.kt`/
`ApiClient.kt`): тело `{team_id: int, points: [...]}`, до **500** точек на батч;
поля точки `id, segment_id, lat, lon, accuracy, altitude?, vertical_accuracy?,
gps_time_ms, trusted_ms?, elapsed_at, boot_count?`; ответ `{"accepted": [id, …]}`.

- **`accepted` = все присланные id.** С `ignore_conflicts=True` точка либо
  вставлена впервые, либо уже есть — для ретрай-цикла клиента и то и другое
  «успех» (ему нужно лишь подтверждение, что можно прекратить слать). Подтверждаем
  весь батч, а не только свежевставленные строки.
- **Валидация all-or-nothing (400 на плохой батч), не частичный приём.**
  Малформный / вне-диапазона point — баг клиента, который должен громко всплыть
  400, а не быть наполовину проглочен. Сериализатор — обычный вложенный, с
  границами полей: `lat ∈ [-90, 90]`, `lon ∈ [-180, 180]`, `min_value=0` на
  физически неотрицательных (`accuracy`/`gps_time_ms`/`elapsed_at`/`trusted_ms`/
  `vertical_accuracy`/`boot_count`), `altitude` **без** нижней границы (может быть
  ниже уровня моря), `id`/`segment_id` `min_length=1`, `points` `max_length=500`.
- **Поток:** резолв published-гонки (404) → валидация тела (400) → проверка, что
  команда в этой гонке (`category2__race_id=race_id`, иначе 404) → `install_id` из
  `request.app_meta` → `bulk_create(ignore_conflicts=True)` → 200 `{"accepted":
  […все id…]}`. Пустой `points` → `bulk_create([])` (no-op), ack `[]`.

## Загрузка взятий КП (`POST /app/race/<id>/marks/`)

Серверный приёмник **взятий КП** (отметок участников на контрольных точках),
которые Android-приложение шлёт батчами в фоне. Эндпоинт — почти клон
`/track/`: та же база `AppAPIView`, тот же **build-HMAC-only** конверт (без
bearer-токена — отметки идут с телефонов **участников**, не админов), тот же
троттл `mobile-write` (60/min), тот же client-UUID-PK + `bulk_create`. Граница
доверия та же, что у read/`track`: валидная сборка может запостить отметку на
**любой** `team_id` гонки; `team_id`/`source_install_id` спуфятся, принимаются
как есть, но подписанное тело делает их tamper-evident. `source_install_id`
читается из **подписанного тела** (ключ провенанса по контракту), а заголовок
`X-Install-Id` остаётся для `AppInstall`-статистики.

**Главное отличие от `/track/`: `Mark` не immutable.** `TrackPoint` строго
неизменяем (`ignore_conflicts` — повторный id молча пропускается), а `Mark`
**обогащается-апсёртится** (`update_conflicts`). Клиент **намеренно пере-шлёт тот
же `id`**, когда данные доезжают посреди запроса: GPS-фикс (`attachLocation`) или
новый участник (`addMember`) приходят после сериализации DTO, но до пометки
строки как загруженной — первый POST несёт `location=null`/частичный `present[]`,
а второй с **тем же `id`** — обогащённый payload. Pre-filter-and-skip **навсегда
потерял бы** поздний GPS/состав, поэтому повтор `id` обязан **мержить**.
Обогащение **монотонно** (location `null→значение`, `present` только растёт,
`complete` `false→true`; момент-времени и `cp_*` стабильны), поэтому слепой
**last-write-wins** апсёрт скаляров корректен и проще всего — без полевой логики
«только если полнее».

**Две нормализованные модели** (`models.py`, FK cross-app в `website`, как у
`apps.race`/`TrackPoint`):

- **`Mark`** — одна строка на отчёт о взятии, **PK = клиентский UUID** (`id`,
  ключ идемпотентности). Плоские поля: провенанс (`team`/`race`/
  `source_install_id`), КП (`checkpoint_id` — **plain int, не FK**, чтобы
  неизвестная КП всё равно принялась; `method` `nfc`/`photo`; `cp_code` hex
  кода КП и `cp_nfc_uid`, оба `blank` под будущий `photo`), мнение клиента
  (`expected_count`/`complete`), серверный `verified`, времена момента
  (`trusted_ms?`/`wall_ms`/`elapsed_at?`/`boot_count?`) и 7 плоских колонок
  `loc_*` (вся `location` может быть `null`). **Нет `updated_at`** — но в отличие
  от `TrackPoint` строка не immutable (обогащается-апсёртится); `updated_at` всё
  равно нет, и модель сознательно **не** в `versioning.py` (никто не читает
  версию с `Mark` — нечего держать свежим). `created_at` (`auto_now_add`)
  **исключён** из `MARK_UPDATE_FIELDS`, поэтому апсёрт сохраняет исходное время
  вставки.
- **`MarkPresent`** — одна строка на присутствующего участника, FK
  `related_name="present"`, с `unique_together("mark", "number_in_team")`.
  Нормализована (а не JSON-колонка), чтобы состав был запрашиваемым для будущей
  (вне области) задачи скоринга/переатрибуции. Идентичность — **физический чип**
  (`nfc_uid`/`code`), сервер резолвит `uid/code → участник` по своему пулу, не
  доверяя клиентскому `number`. Строка `{nfc_uid: null, number: 0}` — **сентинел**
  «снимка нет» (учтённый, но неснятый участник).

**`verified` — серверный, считается при приёме** одним пер-батч префетчем
тег-bid'ов гонки (`_bids_by_checkpoint`, без запроса на каждую отметку): отметка
верифицирована ⇔ `sha256(bytes.fromhex(cp_code))[:16]` совпадает с
`CheckpointTag.bid` заявленной `checkpoint_id` — доказательство физического
скана NFC-кода КП. На проводе `cp_code` — **hex** 16 байт кода, поэтому проверка
`sha256(bytes.fromhex(cp_code)).hexdigest()[:16] == tag.bid`. Неизвестная КП /
несовпадение / не-hex `cp_code` (напр. пустой у будущего `photo`) →
`verified=False`, **строка всё равно сохраняется**. Запрос `.exclude(bid="")`
дропает несобранные теги (`bid` по умолчанию `""`), зеркалит legend-view.

**Идемпотентный merge (НЕ strict-immutable).** Механизм:
`Mark.objects.bulk_create(objs, update_conflicts=True, unique_fields=["id"],
update_fields=MARK_UPDATE_FIELDS)` (Postgres `ON CONFLICT (id) DO UPDATE` — один
атомарный стейтмент, без pre-filter `SELECT`), затем
`MarkPresent.objects.bulk_create(present_objs, ignore_conflicts=True)` для **всех**
отметок батча (аддитивно: `unique_together` пропускает уже сохранённые слоты).
`accepted` эхом возвращает **все** присланные id.

- **In-batch де-дуп по `id` (единственная gotcha `update_conflicts`).** В отличие
  от `ignore_conflicts`, один `INSERT … ON CONFLICT (id) DO UPDATE`, содержащий
  **один и тот же `id` дважды**, кидает Postgres `CardinalityViolation` («cannot
  affect row a second time») → 500. Реальный клиент дублей в батче не шлёт (его
  `unuploaded*`-запрос отдаёт distinct PK), но малформный body не должен 500-ить.
  Поэтому view **де-дупит `marks` по `id` до сборки объектов, оставляя последнее
  вхождение** (консистентно с last-write-wins); `accepted` по-прежнему эхает
  изначально присланные id. `MarkPresent` не затронут — он на `ignore_conflicts`,
  который терпит in-batch дубли.

**Поток** (`MarkUploadView`): резолв published-гонки (404) → валидация тела
(400, all-or-nothing) → проверка, что команда в гонке (`category2__race_id`,
иначе 404) → построить `bids_by_cp` и посчитать `verified` → **де-дуп батча по
`id`** + flatten `location` в 7 `loc_*` (`location is None` → все `loc_*=None`) +
сборка `MarkPresent` из `present[]` → `transaction.atomic()`: **апсёрт** `Mark`
(parent до child, FK-порядок) затем **аддитивная** вставка `MarkPresent` →
200 `{"accepted": [все присланные id]}`. Пустой `marks` → ранний ack `[]` (запрос
тег-bid'ов пропускается).

**Контракт провода** (точно совпадает с `scratch/UPLOAD.md` и клиентским
`MarkDtos.kt`/`ApiClient.uploadMarks`): тело `{team_id, source_install_id, marks:
[…]}`, до **500** отметок; ответ `{"accepted": [id, …]}`. Сериализаторы
(`MarkUploadSerializer`/`MarkSerializer`/`PresentMemberSerializer`/
`MarkLocationSerializer`) переиспользуют `FiniteFloatField` (отбивает NaN/inf),
границы зеркалят `TrackUploadSerializer`. **400-not-500 дисциплина:**
32-битные `IntegerField`-колонки (`checkpoint_id`/`expected_count`/`number`/
`number_in_team`/`boot_count`) несут `max_value=2147483647`, а BigInt-поля
(`trusted_ms`/`wall_ms`/`elapsed_at`/`loc_gps_time_ms`/`loc_elapsed_at`) —
`2**63-1`; перепутать каплю нельзя (2⁶³ в `boot_count` всё ещё 500-ит).
`location` — `required=False` (load-bearing: клиент сериализует с kotlinx
`encodeDefaults=false`, отметка без фикса **опускает** ключ, а не шлёт
`location: null`). `method` — `ChoiceField(["nfc","photo"])`: иное значение → 400.

**Out-of-`versioning.py` посыл — тот же, что у `TrackPoint`.** Никто не читает
версию с `Mark`/`MarkPresent`; они не трогают ETag/`sync`-машинерию.

## Загрузка фото-кадров (`POST /app/race/<id>/mark/<mark_id>/photo/<frame_id>`)

Приёмник **одного сырого JPEG-кадра** для отметки с `method="photo"` — клиент
шлёт по одному запросу на кадр, у одной отметки их может быть несколько.
Организаторское чтение/рендер кадров — отдельная будущая задача, здесь вне
области. Гейт — тот же, что у `/track/`/`/marks/`: **только подпись сборки**
(`AppAPIView`, `[SignedAppPermission]`), без bearer-токена; троттл-скоуп
`mobile-photo` (`120/min`).

**Тело — бинарное, не JSON.** Вью читает `request.body` напрямую и никогда не
трогает `request.data` (дефолтные парсеры DRF не умеют `image/jpeg`, обращение
к `.data` кидает `415`). `SignedAppPermission` уже читает `request.body` до
вью (Django буферизует тело), поэтому подпись по сырым байтам работает без
изменений.

**Модель `MarkPhoto`** (`models.py`, дочерняя FK `mark → Mark` CASCADE,
`related_name="photos"`): файл кладётся по детерминированному,
неугадываемому пути `mark_photos/<mark_id>/<frame_id>.jpg` (два клиентских
UUID) через `FileField` на персистентном томе `/app/media`. **Без поля
`race`** — `race_id` несёт `mark.race_id`, а `race_id` из URL сверяется с
отметкой через `Mark.objects.get(pk=mark_id, race_id=race_id)` (несовпадение
→ `404`). **Нет `updated_at`, сознательно не в `versioning.py`** — тот же
immutable-прецедент, что у `TrackPoint`.

**Идемпотентность по `(mark, frame_id)`**: `unique_together` плюс
предварительный `exists()`-чек схлопывает повтор в `200`; конкурентный дубль,
проскочивший мимо чека, ловится через `IntegrityError` на констрейнте и тоже
подтверждается `200` (с удалением только что записанного файла, чтобы редкая
гонка не текла storage).

**Защита от файла-сироты.** Файл пишется `image.save()` **до** коммита строки,
поэтому крэш в этом окне оставляет канонический файл без строки. На повторе
вью делает `storage.exists()`/`storage.delete()` по каноническому пути перед
записью — иначе storage засуффиксил бы новый файл `_XXXX`, сломав
детерминированность пути.

**Валидация `frame_id`.** `frame_id` подставляется в файловый путь, поэтому
вью проверяет его по `^[A-Za-z0-9_-]{1,64}$` и отвечает `400` до обращения к
storage — но URL-конвертер `<str:frame_id>` (`[^/]+`) уже отбивает пустой или
содержащий `/` сегмент на уровне роутинга `404`-ом раньше вью; гвард вью
срабатывает только на одиночном сегменте с недопустимыми символами (напр.
`a.jpg`). `mark_id` отдельного гварда не требует — хостильное значение просто
не матчит ни одну строку `Mark` (`404`).

**Поток**: `Race` `404` (только published) → `Mark` `404`
(`pk=mark_id, race_id=race_id`) → charset `frame_id` `400` → пустое тело `400`
→ превышение размера `413` (`PHOTO_MAX_BYTES = 10 МБ`) → идемпотентный `200` →
сохранение `201` / `IntegrityError` → `200`.

**`DATA_UPLOAD_MAX_MEMORY_SIZE = 12 МБ`** (`settings.py`) пришлось поднять
выше дефолта Django (2.5 МБ) — лимит проверяется в момент первого чтения
`request.body`, которое происходит **внутри** `SignedAppPermission`, до вью:
кадр меньше нового 10-МБ потолка, но больше старого дефолта 2.5 МБ, иначе
упал бы в непрозрачный `RequestDataTooBig` (`400`) раньше явного `413`. Правка
**глобальная** (любой не-multipart POST теперь может забуферить до 12 МБ), но
принята, т.к. nginx и так режет тела на `client_max_body_size 50m`.

**URL без завершающего слэша** (в отличие от остальных `/app/`-маршрутов) —
подписанная канонная строка включает `full_path` запроса, и она должна
совпасть с клиентским байт-в-байт.

Не зарегистрирован в admin — как и `Mark`/`TrackPoint`/`MarkPresent`
(зарегистрированы только stats-модели `AppInstall`/`AppAuthFailure`).

**Кадр раньше родительской отметки → `404`, это безопасно по контракту**:
клиент начинает выгрузку кадров отметки только после того, как сама отметка
уже подтверждена (`uploaded* == 1`), а `404` на кадре трактует как
временный — повторяет на следующей синхронизации.

## Что обновляется

Приложение держит локальную копию справочников и в фоне поддерживает её
актуальной:

- **список гонок** — какие гонки есть, их статус/флаги;
- **команды гонки** — состав участников;
- **легенда гонки** — контрольные точки; легенда **запертых** КП зашифрована и
  расшифровывается приложением **офлайн только после физического сканирования**
  NFC-кода КП (см. раздел **Шифрование легенды**). Сырой `nfc_uid` на провод не
  уходит; вместо него — единый идентификатор `code`. На каждый физический тег
  отдаётся запись в `tags`: **идентичность** (`bid → checkpoint_id`, всегда, и для
  открытых КП тоже) и — только у запертых КП — **конверт-расшифровка** (`iv`/`ct`).

## Шифрование легенды (запертые КП)

Легенда **запертой** КП (`Checkpoint.is_legend_locked=True`) скрывает
`cost`/`description` за конвертным шифрованием, чтобы приложение могло
расшифровать их **только офлайн и только после физического сканирования NFC-кода
этой КП**. Шифрование **выборочное** (флаг на каждую КП), и один NFC-код может
открывать **настраиваемое подмножество** КП (пересечения разрешены).

**Модель угроз:** противник — само приложение в руках участника. Утечка БД —
**вне области**: `cost`/`description` могут лежать в БД в открытом виде. Поэтому
**at-rest-шифрования и мастер-ключа нет** — шифруется только то, что уходит с
сервера.

### Схема (конвертное шифрование, AES-256-GCM + HKDF)

Примитивы — `crypto.py` (`seal`/`unseal`/`derive_wrap_key`; `unseal`, а не
`open`, чтобы не затенять builtin и не ронять flake8).

1. Каждая запертая КП запечатывается собственным случайным `content_key`
   (32 B) в модель `CheckpointSecret` (`O2O → Checkpoint`,
   `related_name="secret"`): `enc_blob = {"iv","ct"}` — AES-GCM от
   `{cost, description}`, `aad = str(cp.id)`. `id`/`number`/`type` остаются
   публичными. Секрет существует **только** у запертых КП.
2. Каждый `CheckpointTag` несёт случайный `code` (16 B), записываемый в
   user-память NFC-метки **отдельным тулингом** (вне этого кода) для **всех**
   тегов — и запертых, и открытых, и `bid = sha256(code).hexdigest()[:16]`
   (16 hex-символов; мобильный клиент обязан считать его идентично).
   `bid → checkpoint_id` — это **идентичность**: офлайн-распознавание
   `cp_id` для любой КП. Дополнительно у тега запертой КП есть пер-тег
   `bundle_blob = {"iv","ct"}` — AES-GCM от `{cp_id: content_key}` по
   набору `unlocks` тега, ключ обёртки `= HKDF(code)`, `aad = bid`. Пустой
   `unlocks` ⇒ тег открывает свою же КП (`[tag.checkpoint]` по умолчанию в
   `build_bundle`). У открытого тега `bundle_blob` нет (`iv`/`ct` → `None`).
3. Офлайн-поток приложения: прочитать `code` → `bid = sha256(code).hexdigest()[:16]` →
   найти запись по `bid` в `tags` → `checkpoint_id` даёт `cp_id` (идентичность). Если у
   записи есть `iv`/`ct` (запертая КП), `HKDF(code)` расшифровывает бандл →
   `{cp_id: content_key}` → расшифровать `enc` каждой запертой КП.

Конверт (косвенность через `content_key`) и даёт «код → подмножество КП» с
пересечениями (одна КП достижима из нескольких кодов, один код открывает
несколько КП) без пере-шифровки КП под каждый код.

### Хранение и консистентность

Шифртекст и бандлы **предвычислены и хранятся в БД** (так ETag стабилен).
`content_key`/`code` хранятся сырыми (БД доверенная). Согласованность блобов
держат **сигналы** (`signals.py`): `post_save(Checkpoint)` пере-печатывает
секрет, а при переключении флага запертости пересобирает бандлы
`cp.tags.all() ∪ cp.unlocked_by.all()`; `post_save(CheckpointTag)` и
`m2m_changed(unlocks)` пересобирают бандл тега. Защита от рекурсии:
`post_save(CheckpointTag)` рано выходит, когда `update_fields` ровно
`{"code","bid","bundle_blob","updated_at"}` (sentinel сервисного слоя), а
`m2m_changed`-путь огорожен thread-local-флагом.

Массовые операции **обходят сигналы**, поэтому есть команды:

- `manage.py rebuild_legend_crypto [--race <id>] [--regenerate-codes]` —
  бэкфилл после миграции + ремонт (пере-печатать все запертые КП, пересобрать
  все бандлы); `--regenerate-codes` форсит новые `code` (пере-провижининг).
- `manage.py export_legend_codes --race <id>` — печатает `nfc_uid / номер КП /
  code(hex)` для записи кодов в метки.

Bulk-экшены админки «Запереть/Открыть легенду» и «Пересобрать бандл» **обязаны
итерировать и `save()`/звать сервис**, а не `queryset.update()` — иначе сигналы
пропускаются, `CheckpointSecret` не создаётся, и сериализатор отдаёт `cost`/
`description` в открытом виде.

### Что отдаётся

`LegendCheckpointSerializer` ветвится: запертая КП → `{id, number, type, color, enc}`;
открытая → `{id, number, type, color, cost, description}`. `color` — не секрет,
отдаётся в обеих ветках; приложение само маппит токен (`red`, `blue`, …, `""`) на
свою палитру. `TagSerializer` отдаёт на **каждый** тег `{bid, checkpoint_id, check_method}`
(идентичность — всегда, и для открытых КП) и `iv`/`ct` из `bundle_blob` (только у
запертых КП; у открытых → `None`). **`tag_hash` убран** — офлайн-идентичность
теперь `bid → checkpoint_id` для любого тега; путь скоринга при онлайн-сканировании живёт
в `api` и матчит по `nfc_uid` (без изменений).

```json
{
  "race": 42,
  "checkpoints": [
    {"id": 153, "number": 1, "type": "kp", "color": "red", "enc": {"iv": "...", "ct": "..."}},
    {"id": 154, "number": 2, "type": "kp", "color": "blue", "cost": 4, "description": "Дерево у дороги"}
  ],
  "tags": [
    {"bid": "a1b2...", "checkpoint_id": 153, "check_method": "offline", "iv": "...", "ct": "..."},
    {"bid": "c3d4...", "checkpoint_id": 154, "check_method": "offline"}
  ]
}
```

## Браслеты участников (`member_tags`)

`GET /app/race/<id>/member_tags/` отдаёт пул NFC-браслетов участников —
идентичность `number → nfc_uid`, по которой приложение офлайн распознаёт скан
браслета на КП. Сериализатор отдаёт ровно `{number, nfc_uid}` (без `id`/
`last_seen_at`). Тот же read живёт в публичном `api`-эндпоинте
`GET /api/member_tag/` — он **остаётся без изменений**; здесь read лишь
*добавлен* в мобильное приложение под его подписанный, ETag-кешируемый,
пер-гоночный контракт.

**Пер-гоночный URL — задел на будущее.** `race_id` принимается, но **сегодня не
используется для фильтрации**: чип-пул **общий** (один физический набор
браслетов переиспользуется между гонками). URL пер-гоночный, чтобы зарезервировать
хук под будущую возможность «свой набор чипов на гонку» (см. `model Tag` — FK на
гонку пока нет).

**Окно, привязанное к данным.** Отдаётся не весь пул, а
`last_seen_at >= MAX(last_seen_at) − 30 дней` — окно считается от **последнего
скана**, а не от стенных `now()`. У простаивающей гонки набор идеально стабилен;
пол окна сдвигается только реальной скан-активностью. Пустой / ни разу не
сканированный пул (`MAX(last_seen_at) is None`) отдаёт весь пул. Общий helper
`active_member_tags()` кормит **и** view, **и** отпечаток — тело и ETag не могут
разойтись.

**Carve-out `touch` / `updated_at`.** Отпечаток `member_tags_version()` строится по
**фактически отдаваемым значениям полей** — `blake2b` над JSON-кодированием списка
`[[id, number, nfc_uid], …]`, отсортированного по `id`. Хеширование значений (а не
только `MAX(updated_at)|COUNT`) гарантирует, что **равночисленная замена** — один
чип уходит за 30-дневный пол, другой входит — и правка поля (перенумеровка, смена
UID) при конкурентном провижининге оба меняют отпечаток. Поле `Tag.updated_at`
(`auto_now`) двигается только на **провижининг-правках** (добавить / переномеровать /
снять браслет), но **не** на сканах: `MemberTagTouchView` сознательно делает
`save(update_fields=["last_seen_at"])` **без** `"updated_at"` — намеренное
исключение из «update_fields discipline» в `CLAUDE.md`, чтобы каждый тап браслета
не вызывал пересинхронизацию посреди гонки. Скан **сам по себе** ETag не двигает;
ETag всё же может сдвинуться (day-scale, не per-scan), когда скан-активность
достаточно продвинет `MAX(last_seen_at)`, что сменится членство в окне.

**Версия — в манифесте `sync`.** В отличие от `races` (см. ниже),
`versions.member_tags` **включена** в пер-гоночный `sync`: эндпоинт отдаётся по
пер-гоночному URL, и одного синк-опроса гонки достаточно, чтобы узнать, надо ли
перекачивать пул браслетов.

## Принцип синхронизации: полный payload + условный GET (ETag / If-None-Match)

Мы **сознательно не делаем дельта-синк** (`?since=<ts>`, присылать только
изменившиеся строки). На наших объёмах (десятки КП, сотни команд на гонку) он не
окупается: требует `updated_at` на всех моделях **и** тумбстоунов удалений
(удалённая строка не присылает изменений — без отдельной пометки клиент о ней не
узнает). Это лишняя схема синхронизации.

Вместо этого каждый ресурс отдаётся **целиком**, а дешевизну фонового опроса даёт
встроенный в HTTP механизм условных запросов — `ETag` + `If-None-Match`.

### ETag — отпечаток версии ресурса

На каждый ответ сервер вешает заголовок `ETag` — строку-«версию» текущего
содержимого. Правило одно: **тело изменилось → ETag другой; тело то же → ETag
тот же.**

Считается дёшево, без сериализации данных:

- **teams**: `hash(MAX(Team.updated_at), COUNT(*))` по командам гонки **плюс**
  агрегаты по участникам (`Athlet`) и категориям (`Category`) — в ответ teams
  встроен и список категорий гонки, поэтому переименование/реордер/добавление/
  удаление категории тоже двигает версию (`MAX(Category.updated_at)` +
  `COUNT`). Добавили команду → вырос `COUNT`; отредактировали → сдвинулся
  `MAX(updated_at)`.
- **legend**: `MAX(updated_at), COUNT` по `Checkpoint` **плюс** по
  `CheckpointSecret` и `CheckpointTag` того же hidden-исключённого набора,
  поэтому запирание/отпирание КП, пере-шифровка и правка
  тега двигают версию. **Build-независим** — без `key_id` (см. **Статус** ниже).
- **races**: аналогично по `MAX(updated_at), COUNT` соответствующих строк.

ETag оборачивается в кавычки (часть синтаксиса по RFC) и сравнивается
байт-в-байт: `ETag: "a1b2c3"`.

### Поток запросов

**1. Первый запрос — у клиента нет кеша:**

```
→ GET /app/race/8/teams/
   X-App-Sig: …

← 200 OK
   ETag: "a1b2c3"
   {"race": 8, "categories": [ …категории гонки… ], "teams": [ …весь список… ]}
```

Клиент сохраняет локально тело **и** строку `"a1b2c3"`.

Блок `categories` едет **внутри** ответа teams (отдельного эндпоинта нет):
каждая категория — `id`/`code`/`short_name`/`name`/`order`, отсортированы по
`order, id`. Включаются **все** категории гонки, в том числе `is_active=False`
(команда может ссылаться на снятую категорию — её `category2`-id должен
резолвиться); само поле `is_active` не отдаётся. Изменение списка категорий
двигает тот же `versions.teams` / ETag teams — отдельного `versions.categories`
нет.

**2. Фоновый опрос — ничего не поменялось:**

```
→ GET /app/race/8/teams/
   If-None-Match: "a1b2c3"      ← эхом возвращаем запомненный ETag
   X-App-Sig: …

← 304 Not Modified
   (тело пустое)
```

Сервер посчитал текущий ETag, он совпал → `304` **без тела**. Копеечный опрос:
сотни байт заголовков, ноль данных, ноль парсинга JSON на телефоне. Главное —
при 304 сервер **даже не сериализует** список: посчитал ETag (один SQL
`aggregate`) и вышел.

**3. Кто-то добавил/изменил команду:**

```
→ GET /app/race/8/teams/
   If-None-Match: "a1b2c3"
   X-App-Sig: …

← 200 OK
   ETag: "d4e5f6"               ← новая версия
   {"race": 8, "categories": [ … ], "teams": [ …обновлённый список… ]}
```

ETag не совпал → полный свежий payload + новый ETag. Клиент перезаписывает копию
и запоминает `"d4e5f6"`.

### Почему удаления «просто пропадают»

При смене ETag отдаётся **весь актуальный список**. Удалённой команды в новом
списке нет → клиент затирает старую копию новой, строка исчезает сама. Никаких
тумбстоунов — отсутствие дельта-синка и есть отсутствие схемы синхронизации.

### Серверная сторона (псевдокод)

```python
def get(self, request, race_id):
    current_etag = self.compute_etag(race_id)        # hash(MAX(updated_at), COUNT)
    if request.headers.get("If-None-Match") == current_etag:
        return Response(status=304)                  # данные НЕ сериализуем
    data = self.serialize(race_id)
    resp = Response(data)
    resp["ETag"] = current_etag
    return resp
```

## Взаимодействие с подписью

- `If-None-Match` **не входит** в канонную строку
  (`method\nfull_path\nts\nsha256(body)`) и не должен: он не меняет смысл запроса.
  В худшем случае при рассинхроне клиент получит `200` вместо `304` и просто
  перекачает данные — безопасность не страдает, `signing.py` трогать не нужно.
- Порядок проверок: **подпись → ETag**. `SignedAppPermission` отрабатывает как
  обычно (невалидная подпись → нейтральный `403`); `304` возвращается **только
  после** успешной проверки подписи, на валидном запросе. 304 — это не ошибка и
  не разновидность 403.
- Используем **сильные** ETag (без префикса `W/`): отпечаток привязан к данным.

## Два сервера и хэндофф

В бою два сервера: **облачный** (в интернете) и **локальный** (в LAN на гонке).
Данные правят на **обоих** — например, во время гонки админ на локальном
уменьшает состав команды (участник не приехал). Опасность: телефон ловит
интернет, синкается с облака и **возвращает** прежний состав → команде не хватает
людей для отметки на КП.

Это **не** задача свежести кеша — это **источник правды и разрешение
конфликтов**. ETag/`data_version` отвечают на «актуальна ли моя копия *этого
сервера*», но **не** на «чьи данные главнее, когда серверы разошлись».

### Версия конфликт не решает (и может усугубить)

Версия арбитрит конфликт, только если её выдаёт **один** авторитет. У нас два
независимых писателя — счётчики версий у серверов свои, **сравнивать их между
origin нельзя**. Правило «побеждает бóльшая версия» → телефон увидит у облака
версию повыше (там состав не трогали) и вернёт старое число. Конфликт двух
одновременных писателей надёжно решается не версией, а тем, чтобы **их не было
одновременно**.

### Правило: один авторитет на гонку в каждой фазе

```
До гонки:    источник правды — облако.  Локальный — зеркало (тянет с облака).
Во время:    источник правды — ЛОКАЛЬНЫЙ. Телефон читает операционные данные
             этой гонки ТОЛЬКО с локального. Облако к ней не допускается.
После:       сверка локальный → облако (backend-задача, телефона не касается).
```

```
   облако ──(сид перед гонкой)──▶ локальный
                                     │  во время: правки админа + отметки КП
                                     ▼
   облако ◀──(сверка после гонки)── локальный
```

Во время гонки телефон **вообще не тянет состав этой гонки из облака**, поэтому
перезаписать локальную правку нечем — писатель один, мержить нечего.

### Как это выражено в данных

1. **Origin-тег на каждой кешированной записи** — телефон помнит, откуда взял
   данные: `source: "local"` / `source: "cloud"`.
2. **Лиз авторитета в payload.** Локальный сервер помечает гонку как оперируемую
   им — **со сроком**, а не просто флагом:

   ```json
   { "data_source": "local", "lease_expires_at": 1718200000 }
   ```

   где `lease_expires_at` ≈ конец гонки + запас. Телефон, увидев в LAN такой
   сервер с этой гонкой, **пинит** её на локальный: состав/легенда и ETag —
   только локального.
3. **Правило «не понижать через границу origin»:** запись с `source:"local"` для
   активного лиза **никогда** не перезаписывается данными `source:"cloud"`, даже
   если телефон случайно дёрнул облако.

### Возврат на cloud: по истечению лиза, а не по потере связи

Пин — это **лиз, а не залипающая защёлка**. Освобождает его **время/фаза, а не
связь**. Триггер «вернуться на cloud, когда локальный недоступен» — **ошибка**:
телефон вышел из Wi-Fi на 5 минут → откат на cloud → cloud вернул прежний состав.

- **Транзиентный обрыв во время гонки** → лиз ещё жив → телефон работает с
  последней локальной копией из кеша, в облако за этой гонкой не ходит.
- **Гонка кончилась, локальный выключили** → лиз сам истёк → телефон
  автоматически возвращается на cloud, без «чистого» сигнала от мёртвого сервера.

**Heartbeat:** каждый успешный синк с локальным **продлевает**
`lease_expires_at`. Телефон уходит на cloud, только когда **и** не может продлить
(локальный недоступен), **и** последний известный срок истёк.

**Быстрый путь штатного завершения:** если телефон дотянулся до локального, а тот
говорит `data_source:"cloud"` / `live:false` (или убрал гонку) — пин снимается
**сразу**.

Триггеры возврата на cloud:

| Событие | Возврат на cloud? |
|---|---|
| Локальный недоступен, лиз ещё жив | **Нет** (используем кеш локального) |
| Локальный недоступен, лиз истёк | Да (авто) |
| Локальный отвечает `live:false` / `data_source:"cloud"` | Да (сразу) |
| Лиз истёк, но локальный ещё отдаёт live и продлевает | Нет (heartbeat продлил) |

Лиз — **per-race**: одна гонка может быть live на локальном, другая — обычная
облачная.

**Caveat:** между истечением лиза и сверкой `локальный → облако` есть окно, где
телефон может показать ещё несведённые облачные данные. Ставь `lease_expires_at`
**с запасом на сверку** — лиз должен пережить момент, когда облако догонит
локальные правки (либо постгоночная сверка пушит в облако до истечения лиза).

### Где остаётся ETag / `data_version`

Там же — **внутри одного origin**, для свежести: ETag локального проверяет
актуальность локальной копии, ETag облака — облачной. Это по-прежнему **два ETag,
по origin** (см. выше). Межсерверный арбитр — флаг авторитета
(`data_source`/`lease_expires_at`), **а не** число версии.

## Манифест: эндпоинт `sync`

Фоновый опрос трёх ресурсов (`races`/`teams`/`legend`) по отдельности — это три
подписанных запроса на каждый тик. Чтобы схлопнуть их в **один**, есть лёгкий
эндпоинт-манифест:

```
GET /app/race/<id>/sync/        → mobile:sync
```

Отдаёт по строке-версии на каждый ресурс гонки **плюс** флаг авторитета — то есть
объединяет две вещи, о которых шла речь выше: указатели свежести (для ETag-логики)
и лиз хэндоффа (для выбора сервера):

```json
{
  "race": 8,
  "data_source": "local",
  "lease_expires_at": 1718200000,
  "versions": {
    "teams":  "a1b2c3",
    "legend": "d4e5f6",
    "member_tags": "9a8b7c"
  }
}
```

`versions.member_tags` глобальна (пул браслетов общий), но включена в манифест,
потому что эндпоинт пер-гоночный — в отличие от `races`, чей глобальный список
клиент пробует своим conditional GET до выбора гонки.

Поток в фоне:

1. Телефон дёргает **один** `sync` (дёшево: пара SQL-`aggregate`, без сериализации
   данных).
2. Сверяет `versions[*]` с тем, что закешировал; тянет полным запросом **только**
   изменившиеся ресурсы.
3. Из того же ответа берёт `data_source`/`lease_expires_at` — продлевает лиз и
   решает, с каким сервером вообще работать (см. **Два сервера и хэндофф**).

Так фоновый тик — это `1` запрос вместо `3`, а полные `GET teams/legend` уходят
только по факту изменения. `data_source`/`lease_expires_at` приходят из манифеста
**того сервера, к которому обратились**: локальный объявляет свой лиз, облако — что
гонка облачная.

Почему манифест, а не просто ETag/304 на каждом ресурсе: на трёх ресурсах выигрыш
по числу запросов скромный, но манифест даёт **консистентный снимок** версий +
авторитета за один round-trip и одно место для `lease_expires_at`. Базовый
ETag/`If-None-Match` на ресурсах остаётся (см. выше) — `sync` его не заменяет, а
надстраивается: манифест говорит «что качать», ETag на ресурсе остаётся защитой от
гонки между манифестом и полным GET.

## Эндпоинты

| Метод | Путь | Имя | Отдаёт |

| GET | `/app/race/<id>/member_tags/` | `mobile:member_tags` | пул браслетов участников (`number → nfc_uid`); см. **Браслеты участников** ниже |
| GET | `/app/race/<id>/sync/` | `mobile:sync` | манифест: версии ресурсов + `data_source`/`lease_expires_at` |

Все — наследники `AppAPIView` (подпись + `AppInstall`-статистика); у `races`,
`teams`, `legend` и `member_tags` — поддержка `ETag`/`If-None-Match` (304), у
`sync` — лёгкий манифест версий для фонового обновления (без 304). Версия `races`
глобальная (`races_version()`, без `race_id`) и сознательно **не входит** в
пер-гоночный манифест `sync` — список гонок клиент пробует напрямую своим
conditional GET (это входная точка приложения, до выбора `race_id`). Версия
`member_tags` тоже глобальная (пул браслетов сегодня общий), но **входит** в
`sync` — она отдаётся по пер-гоночному URL, и приложение должно за один тик
синка узнать, что перекачивать для синкаемой гонки.

> **Статус:** реализованы `mobile:races` (с `ETag`/`If-None-Match` → 304;
> отпечаток — `MAX(Race.updated_at)|COUNT` по published-гонкам, поле
> `Race.updated_at` — миграция `0079`), `mobile:legend` (с `ETag`/`If-None-Match`
> → 304; запертые КП отдают только `enc`, идентичность тегов — в `tags`),
> `mobile:teams` (с `ETag`/`If-None-Match` → 304) и
> `mobile:sync`. Отпечаток `teams` считается по
> `MAX(Team.updated_at)`/`MAX(Athlet.updated_at)`/`COUNT` команд и участников
> **плюс** `MAX(Category.updated_at)`/`COUNT(Category)` по категориям гонки
> (поле `Athlet.updated_at` добавлено, чтобы переименование участника сдвигало
> версию; `Category.updated_at` — миграция `0078` — чтобы правка встроенного в
> ответ списка категорий тоже двигала версию). Отпечаток `legend` —
> `MAX(Checkpoint.updated_at)|COUNT` по **hidden-исключённому**
> набору КП (поле `Checkpoint.updated_at` добавлено миграцией `0077` по той же
> причине, что и `Athlet.updated_at` — ловить правки на месте) **плюс** ещё два
> агрегата `MAX(updated_at)|COUNT` по тому же hidden-исключённому набору:
> `CheckpointSecret` (пере-шифровка / появление-исчезновение `enc` при
> запирании-отпирании) и `CheckpointTag` (`code`/`unlocks`/`bundle`/`check_method`).
> Легенда теперь **build-независимая**: шифртекст и конверты **предвычислены и
> хранятся в БД** (не зависят от секрета сборки), поэтому отпечаток **не** содержит
> `key_id` и две разные сборки получают **один** ETag/`versions.legend` — ротация
> `X-App-Key-Id` больше не пере-качивает легенду. Реализован и `mobile:member_tags`
> (с `ETag`/`If-None-Match` → 304; отпечаток хеширует фактические значения полей
> `(id, number, nfc_uid)` по data-anchored окну `active_member_tags()`;
> скан-`touch` ETag не двигает). Манифест
> `sync` несёт `versions.teams`, `versions.legend` **и** `versions.member_tags`.
> Лиз-хэндофф **застаблен**: `data_source`
> берётся из настройки `MOBILE_DATA_SOURCE` (по умолчанию `"cloud"`),
> `lease_expires_at` всегда `null` — реальный per-race лиз/хэндофф остаётся целевой
> схемой, описанной выше.
